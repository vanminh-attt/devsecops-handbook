Vì sao phải học cả Docker/K8s và kiến trúc mạng? Vì pipeline CI/CD hiện đại không deploy lên "một con server" nữa — nó build ra container image và deploy lên Kubernetes, chạy trong VPC trên cloud. Không hiểu container thì không hiểu mình đang build cái gì; không hiểu K8s thì không hiểu Argo CD deploy đi đâu; không hiểu VPC/subnet thì không đọc nổi sơ đồ hạ tầng như Diagram V2 của bạn. Ba thứ này là mặt đất mà pipeline đứng lên trên.

02Khái niệm nền

DevSecOps là gì — và vì sao "shift-left"

DevOps là văn hóa + bộ công cụ giúp đội Dev và đội Ops làm việc như một: mọi thay đổi code được tự động build, test và deploy liên tục thay vì release thủ công vài tháng một lần. DevSecOps thêm chữ Sec vào giữa: bảo mật không còn là khâu "kiểm tra cuối cùng trước khi lên production" mà được nhúng vào từng bước của vòng đời phát triển phần mềm (SDLC).

Shift-left: nguyên tắc quan trọng nhất

"Shift-left" nghĩa là đẩy việc phát hiện lỗ hổng về càng sớm (càng về bên trái của timeline) càng tốt. Lý do thuần túy là chi phí: một lỗ hổng phát hiện lúc dev đang gõ code sửa mất 5 phút; cũng lỗ hổng đó lọt ra production có thể thành sự cố bảo mật tốn hàng tuần xử lý và thiệt hại thật. Toàn bộ chương 10 là các "trạm kiểm soát" đặt dọc pipeline để làm việc này.

Phân biệt CI / CD / CD

  • Continuous Integration (CI): mỗi lần dev push code, hệ thống tự động build + chạy test + quét bảo mật. Mục tiêu: phát hiện code hỏng/nguy hiểm trong vài phút.
  • Continuous Delivery: mọi bản build đạt chuẩn đều sẵn sàng deploy — nhưng bấm nút deploy vẫn là quyết định của con người (approval gate).
  • Continuous Deployment: tự động deploy luôn không cần người bấm. Thường chỉ áp dụng cho môi trường dev/staging; production hầu như luôn giữ approval gate — đúng như sơ đồ của bạn đang vẽ (non-prod auto sync, prod có sync window + approval).
Bảo mật

Bản thân pipeline cũng là mục tiêu tấn công (supply chain attack — vụ SolarWinds, Codecov là ví dụ kinh điển). Vì thế học DevSecOps không chỉ là "quét code của app" mà còn là bảo vệ chính pipeline: quản lý credentials của Jenkins, ký image, khóa quyền của Argo CD. Toàn bộ chương 11 dành riêng cho chủ đề này: OWASP CI/CD Security Top 10 và chuỗi cung ứng.

03Lộ trình

Lộ trình học 6 giai đoạn (~6–9 tháng)

Đi tuần tự, không nhảy cóc: DevSecOps là tầng trên cùng của tháp, mỗi giai đoạn đứng trên giai đoạn trước. Mỗi giai đoạn có "sản phẩm đầu ra" — làm xong sản phẩm đó mới chuyển giai đoạn, vì đó chính là bằng chứng bạn mang đi phỏng vấn.

Giai đoạn 03–4 tuần

Nền tảng: Linux, Git, mạng, scripting

  • Linux: terminal, quyền file, systemd, SSH — dựng qua WSL2 ngay trên máy Windows của bạn.
  • Git: branching (GitFlow, trunk-based), Pull Request, merge/rebase — vì mọi pipeline đều kích hoạt từ Git.
  • Mạng: DNS, HTTP/HTTPS, TLS, port, firewall, CIDR (bắt buộc để hiểu VPC).
  • Bash bắt buộc + Python để viết automation.
Sản phẩm: một repo GitHub có script Bash tự động hóa việc bất kỳ (backup, parse log), làm việc hoàn toàn qua terminal.

  • Khái niệm pipeline, stage, artifact, runner/agent, webhook.

  • Docker: image/layer, Dockerfile multi-stage, network, volume, compose → chương 05.

CIDR — bắt buộc để hiểu VPC

10.0.0.0/16 nghĩa là: 16 bit đầu cố định (10.0), 16 bit sau tự do → 65.536 địa chỉ IP từ 10.0.0.0 đến 10.0.255.255. Số sau dấu / càng lớn thì dải càng nhỏ: /24 = 256 IP (một subnet điển hình). Trong Diagram V2 của bạn: Ingress VPC là 10.0.0.0/16, Workload VPC là 10.10.0.0/16 — hai dải không chồng lấn để Transit Gateway route được giữa chúng.

05Docker

Docker chuyên sâu

Kiến trúc và khái niệm cốt lõi

Container là một tiến trình Linux bình thường nhưng bị cô lập bằng 2 cơ chế của kernel: namespaces (mỗi container thấy cây tiến trình, mạng, filesystem riêng) và cgroups (giới hạn CPU/RAM). Khác máy ảo: container dùng chung kernel với host nên khởi động trong mili-giây và nhẹ hơn hàng chục lần — nhưng cũng vì dùng chung kernel nên bảo mật container là chủ đề riêng (mục cuối chương).

Vì sao layer quan trọng

Mỗi lệnh trong Dockerfile tạo một layer và được cache. Đặt thứ ít thay đổi (cài dependencies) lên trên, thứ hay thay đổi (copy source code) xuống dưới → build lại chỉ mất vài giây thay vì vài phút. Đây là câu hỏi phỏng vấn kinh điển.

Dockerfile: từng lệnh và multi-stage build

# ---- Stage 1: build (chứa toàn bộ toolchain, sẽ bị vứt đi) ----
FROM node:22-alpine AS build
WORKDIR /app
COPY package*.json ./          # copy riêng file deps trước → tận dụng layer cache
RUN npm ci --omit=dev
COPY . .
RUN npm run build

# ---- Stage 2: runtime (chỉ mang theo thứ cần để chạy) ----
FROM node:22-alpine
RUN addgroup -S app && adduser -S app -G app   # tạo user thường — KHÔNG chạy bằng root
WORKDIR /app
COPY --from=build --chown=app:app /app/dist ./dist
COPY --from=build --chown=app:app /app/node_modules ./node_modules
USER app
EXPOSE 3000
HEALTHCHECK CMD wget -qO- http://localhost:3000/health || exit 1
CMD ["node", "dist/server.js"]

Network và Volume

  • bridge (mặc định): container cùng một network tự phân giải tên nhau (db:5432). Luôn tạo network riêng cho app: docker network create mynet.
  • host: dùng thẳng network của máy host — nhanh nhưng mất cô lập.
  • Volume: dữ liệu sống lâu hơn container. docker volume create pgdata rồi -v pgdata:/var/lib/postgresql/data. Bind mount (-v ./src:/app) dùng khi dev để code sync trực tiếp.

Bảo mật Docker — checklist

  • Chạy bằng user thường (USER app), không bao giờ để mặc định root.
  • Base image nhỏ và có tag cụ thể: alpine, distroless; cập nhật định kỳ.
  • Không nhét secrets vào Dockerfile, ENV hay layer — dùng runtime secrets (Vault, K8s Secret, Secrets Manager).
  • Quét image trước khi push: trivy image myapp:1.2.0 (chương 10).
  • Thêm .dockerignore (loại .git, .env, node_modules) — tránh lộ secrets vào build context.
  • Giới hạn tài nguyên: --memory=512m --cpus=1; filesystem chỉ đọc: --read-only khi có thể.
  • Không bao giờ mount /var/run/docker.sock vào container không tin cậy — tương đương trao quyền root cả host.
  • Ký image bằng Cosign và bật scan-on-push trên registry (Harbor/ECR đều hỗ trợ — sơ đồ của bạn ghi "ECR scan/sign" chính là bước này).
06Kubernetes

Kubernetes chuyên sâu

Kubernetes (K8s) là hệ điều hành của cụm máy chủ: bạn khai báo trạng thái mong muốn bằng YAML ("tôi muốn 3 bản copy app này chạy, có load balancer phía trước"), K8s liên tục so sánh thực tế với mong muốn và tự sửa cho khớp. Nguyên lý "declarative + reconcile" này chính là nền của GitOps/Argo CD ở chương 09.

CI/CD & Jenkins

Pipeline là gì — giải phẫu

Pipeline (đường ống) là một chuỗi bước tự động: mỗi lần bạn đẩy code lên Git, hệ thống chạy lần lượt các bước từ đầu đến cuối mà không cần ai làm thủ công. Hình dung như một dây chuyền sản xuất — nguyên liệu (source code) đi vào một đầu, qua từng trạm (build → test → quét bảo mật → đóng gói → triển khai), ra đầu kia là sản phẩm chạy được trên môi trường thật. Nếu một trạm báo lỗi, cả dây chuyền dừng: không cho hàng lỗi đi tiếp.

Đừng nhầm "pipeline" với "công cụ": Jenkins, GitLab CI, GitHub Actions chỉ là động cơ chạy pipeline. Bản thân pipeline là một file khai báo nằm ngay trong repo (Jenkinsfile, .gitlab-ci.yml, .github/workflows/*.yml) — gọi là Pipeline as Code: pipeline được version và review y như code thường, ai sửa quy trình build/deploy đều để lại dấu vết.

Jenkinsfile DevSecOps hoàn chỉnh (khung sườn để học và tái dùng)

pipeline {
  agent any
  environment {
    IMAGE    = "registry.example.com/myapp"
    TAG      = "${env.GIT_COMMIT.take(8)}"
    REGISTRY = credentials('registry-cred')      // lấy từ Credentials store
  }
  options { timestamps() }

  stages {
    stage('Checkout')        { steps { checkout scm } }

    stage('Secrets scan')    { steps {                // chặn API key/password lộ trong code
      sh 'gitleaks detect --source . --report-format sarif --report-path gitleaks.sarif'
    } }

    stage('Build + Unit test') { steps {
      sh 'npm ci && npm test'
    } }

    stage('SAST') { steps {                           // quét lỗ hổng trong source
      withSonarQubeEnv('sonar') { sh 'sonar-scanner -Dsonar.projectKey=myapp' }
    } }
    stage('Quality Gate') { steps {                   // SonarQube fail → pipeline dừng
      timeout(time: 5, unit: 'MINUTES') { waitForQualityGate abortPipeline: true }
    } }

    stage('SCA - dependencies') { steps {             // thư viện có CVE Critical → dừng
      sh 'trivy fs --scanners vuln --severity HIGH,CRITICAL --exit-code 1 .'
    } }

    stage('Docker build') { steps {
      sh 'docker build -t $IMAGE:$TAG .'
    } }

    stage('Image scan') { steps {                     // image có CVE Critical → không được push
      sh 'trivy image --severity CRITICAL --exit-code 1 $IMAGE:$TAG'
    } }

    stage('Push + Sign') { steps {
      sh '''
        echo $REGISTRY_PSW | docker login registry.example.com -u $REGISTRY_USR --password-stdin
        docker push $IMAGE:$TAG
        cosign sign --yes $IMAGE:$TAG              # ký image — chống supply chain attack
        syft $IMAGE:$TAG -o cyclonedx-json > sbom.json   # tạo SBOM
      '''
    } }

    stage('Update GitOps repo') { steps {             // KHÔNG deploy trực tiếp — chỉ sửa Git,
      sh '''                                          # Argo CD sẽ tự thấy và deploy (chương 09)
        git clone https://git.example.com/platform/gitops-config.git
        cd gitops-config
        sed -i "s|image: $IMAGE:.*|image: $IMAGE:$TAG|" apps/myapp/dev/deployment.yaml
        git commit -am "deploy: myapp $TAG to dev" && git push
      '''
    } }
  }
  post {
    always  { archiveArtifacts artifacts: '*.sarif, sbom.json', allowEmptyArchive: true }
    failure { echo 'Gửi cảnh báo Slack/Teams tại đây' }
  }
}

Runner là nơi code lạ được thực thi ngay cạnh secrets — tâm điểm của nhiều rủi ro trong OWASP CI/CD Top 10 (chương 11). Biến CI, secrets và bảo vệ nhánh — phần DevSecOps quan tâm nhất

  • CI/CD Variables (Settings → CI/CD → Variables): bật Masked để giá trị bị che thành [MASKED] trong log, và Protected để biến chỉ cấp cho pipeline chạy trên protected branch/tag — MR từ nhánh lạ không bao giờ nhìn thấy secret prod. Đây là phòng thủ trực tiếp cho PPE (chương 11).
  • Protected branch cho main/prod: cấm push thẳng, bắt buộc MR + ít nhất 1 approval; thêm CODEOWNERS để mọi thay đổi vào .gitlab-ci.yml phải có platform team duyệt.
  • Environment + protected environment: gắn environment: production cho job deploy — chỉ nhóm được chỉ định mới bấm chạy được, và GitLab lưu lịch sử ai deploy gì lúc nào.
  • OIDC (id_tokens) thay token tĩnh: pipeline xin JWT ngắn hạn rồi đổi lấy quyền AWS/Vault — không còn access key nằm chết trong Variables. Cùng triết lý với IRSA bạn gặp ở chương 06.
  • Biến có sẵn hay dùng: $CI_COMMIT_SHORT_SHA, $CI_REGISTRY_IMAGE, $CI_PIPELINE_SOURCE (push? MR? schedule?), $CI_JOB_TOKEN (token ngắn hạn tự có của mỗi job).

GitHub Actions — viết cho đúng và cho an toàn

GitHub Actions là CI dễ học nhất và là thứ bạn nên áp ngay vào các repo portfolio GitHub của mình. Workflow là file YAML trong .github/workflows/, kích hoạt theo sự kiện (on:), chạy trên runner của GitHub hoặc self-hosted:

name: ci
on:
  push: { branches: [main] }
  pull_request:

permissions: { contents: read }   # hạ GITHUB_TOKEN xuống chỉ-đọc — cấp thêm theo từng job

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<commit-sha-đầy-đủ>      # v4 — pin theo SHA, không dùng tag trôi
      - uses: gitleaks/gitleaks-action@<commit-sha>
        env: { GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }} }
      - run: npm ci && npm test
      - run: docker build -t myapp:${{ github.sha }} .
      - uses: aquasecurity/trivy-action@<commit-sha>     # quét image, fail nếu CRITICAL
        with: { image-ref: "myapp:${{ github.sha }}", severity: CRITICAL, exit-code: 1 }

Checklist an toàn GitHub Actions

  • permissions: contents: read ngay đầu workflow — GITHUB_TOKEN mặc định có thể ghi vào repo; tự hạ xuống chỉ-đọc rồi cấp thêm đúng job cần (vd job release cần contents: write).
  • Pin action theo commit SHA, không theo tag: tag @v4 có thể bị chủ action (hoặc kẻ chiếm account của họ) trỏ sang code độc — chính là vụ tj-actions/changed-files tháng 3/2025 khiến hàng nghìn repo bị rút secrets từ log build.
  • Cảnh giác pull_request_target: trigger này chạy với secrets của repo — nếu lại checkout code từ PR của người lạ thì họ chiếm được secrets (lỗ hổng P-PPE kinh điển, chương 11). Dùng pull_request thường trừ khi hiểu rõ mình đang làm gì.
  • OIDC sang cloud: dùng aws-actions/configure-aws-credentials với role-to-assume — xóa hẳn AWS access key tĩnh khỏi repo secrets.
  • Secrets deploy đặt trong Environment có required reviewers — lên prod phải có người duyệt.
  • Bật Dependabot cho cả github-actions lẫn npm/pip; thêm dependency-review-action để PR kéo vào thư viện có CVE là bị chặn.

"Golden Path": template hoá secure code cho cả tổ chức

Bạn hỏi rất đúng: ngoài pipeline, DevSecOps còn có khái niệm template secure code — dựng sẵn "khuôn" đã an toàn theo mặc định (secure by default) để dev cứ thế điền code vào, không phải tự nhớ bật từng biện pháp bảo mật. Trong ngành gọi mô hình này là Golden Path (hay Paved Road — "con đường đã lát sẵn"): platform/security team làm sẵn con đường an toàn và dễ đi nhất, để lựa chọn an toàn cũng chính là lựa chọn tiện nhất — dev không phải chọn giữa "nhanh" và "an toàn".

  • Cách đẩy (push): code từ IDE → git commit → git push lên nhánh feature → mở Merge/Pull Request. Push hoặc MR là trigger làm pipeline chạy (qua webhook). Nhánh main/prod phải là protected branch: cấm push thẳng, bắt buộc qua MR + review (chương 08, phần secrets & protected branch).
  • Quét code: pipeline chạy tuần tự các trạm bảo mật đã học — Gitleaks (secret) → Semgrep/SonarQube (SAST) → Trivy (SCA + image) → ZAP (DAST ở môi trường test). Mỗi trạm là một gate; gặp lỗ hổng nghiêm trọng thì fail, MR không được merge (chương 10).

GitOps & Argo CD

Triết lý GitOps và kiến trúc Argo CD

GitOps = trạng thái mong muốn của hệ thống (toàn bộ manifest/Helm chart) nằm trong một Git repo; một controller trong cluster liên tục so sánh Git nói gì với cluster đang thế nào và tự đồng bộ. Hệ quả quan trọng với bảo mật:

  • Audit trail miễn phí: mọi thay đổi production là một commit — ai, lúc nào, thay đổi gì, được ai approve (PR review). Đối với hệ thống tài chính như trong sơ đồ của bạn, đây là yêu cầu tuân thủ, không phải tùy chọn.
  • Pull-based: Argo CD trong cluster tự kéo về, CI không cần cầm credentials của cluster.
  • Self-heal + chống drift: ai đó sửa tay trên cluster (kể cả kẻ tấn công) → Argo CD phát hiện lệch so với Git và đưa về trạng thái đúng.
  • Rollback = git revert: quay về trạng thái cũ trong vài giây.
# Cài Argo CD lên cluster lab (minikube/kind)
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
# Lấy mật khẩu admin và mở UI:
argocd admin initial-password -n argocd
kubectl port-forward svc/argocd-server -n argocd 8443:443
# → mở https://localhost:8443 , user: admin

Các thành phần chính: repo-server (kéo Git, render Helm/Kustomize ra manifest), application-controller (so sánh và sync), api-server/UI (giao diện, RBAC, SSO). Trạng thái ứng dụng đọc theo 2 trục: Sync status (Synced / OutOfSync — Git và cluster có khớp không) và Health (Healthy / Progressing / Degraded — app có sống không).

Application — đơn vị làm việc của Argo CD

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-prod
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://git.example.com/platform/gitops-config.git
    targetRevision: prod              # nhánh prod — khớp mô hình dev/test/stg/prod của bạn
    path: apps/myapp/prod
    helm: { valueFiles: [values-prod.yaml] }
  destination:
    server: https://kubernetes.default.svc   # hoặc URL cluster spoke khác
    namespace: prod
  syncPolicy:
    automated:
      prune: true                     # xóa resource không còn trong Git
      selfHeal: true                  # sửa tay trên cluster → tự đưa về theo Git
    syncOptions: [CreateNamespace=true]

App-of-Apps và Hub-and-Spoke — mô hình trong sơ đồ của bạn

  • App-of-Apps: một Application "mẹ" trỏ vào thư mục chứa các Application "con" → thêm app mới = thêm 1 file YAML vào Git, không đụng tay vào Argo CD.
  • Hub-and-Spoke: một Argo CD đặt ở cluster trung tâm (Hub) quản lý deploy cho nhiều cluster đích (Spoke). Trong Diagram V2: Hub EKS nằm ở Shared Services Account, các Spoke là cluster Non-Prod (auto sync) và Prod (sync windows) — Hub được cấp quyền vào Spoke qua EKS Access Entry với least privilege, đúng ghi chú trong sơ đồ. Ưu điểm: một mặt phẳng điều khiển duy nhất, một nơi audit; đổi lại Hub trở thành tài sản trọng yếu phải bảo vệ kỹ (RBAC, SSO, network isolation).
  • Tra thêm: ApplicationSet — generator tự sinh Application cho N cluster/N môi trường từ một template.

Trivy đáng học đầu tiên: một binary duy nhất quét được dependency (lớp 3), image (lớp 4), IaC (lớp 5) và cả secrets (lớp 1). Sau đó SonarQube (lớp 2) vì gần như mọi doanh nghiệp VN đều dùng. Các lớp còn lại thêm dần theo giai đoạn 3 của lộ trình.

Lệnh mẫu từng tool — chạy được ngay

# 1. Gitleaks — quét secrets trong repo và toàn bộ lịch sử Git
gitleaks detect --source . -v

# 2a. Semgrep — SAST nhanh, rule cộng đồng
semgrep scan --config auto .
# 2b. SonarQube — dựng server bằng Docker rồi quét

# 3. Trivy — quét dependency + secrets trong thư mục dự án
trivy fs --scanners vuln,secret,misconfig .

# 4. Trivy — quét container image, fail nếu có CRITICAL (dùng làm quality gate)

Vault — quản lý secrets tập trung (lớp 7)

Vấn đề Vault giải quyết: secrets rải rác trong file .env, biến CI, K8s Secret (chỉ là base64) — không ai biết có bao nhiêu, ai đang dùng, bao giờ xoay vòng. Vault tập trung tất cả về một chỗ, cấp phát động (mỗi lần xin là một credential mới, có hạn dùng), ghi log mọi lần truy cập.

docker run -d --name vault -p 8200:8200 --cap-add=IPC_LOCK hashicorp/vault   # dev mode để học
export VAULT_ADDR=http://localhost:8200
vault kv put secret/myapp db_password=S3cret!          # ghi secret
vault kv get -field=db_password secret/myapp            # đọc secret

Trên K8s, dùng External Secrets Operator hoặc Secrets Store CSI Driver để Pod tự lấy secret từ Vault/AWS Secrets Manager lúc khởi động — manifest trong Git không bao giờ chứa secret thật. Sơ đồ của bạn ghi "Secrets Manager + CSI" là mô hình này với dịch vụ của AWS.

Sigstore — ký keyless và verify ở cổng cluster

Điểm yếu của ký truyền thống: private key phải cất ở đâu đó — và chỗ đó trộm được (chính là cách SolarWinds có chữ ký hợp lệ). Cosign keyless ký bằng danh tính OIDC của pipeline: Fulcio cấp chứng chỉ ngắn hạn theo danh tính đó, mọi chữ ký ghi vào transparency log công khai Rekor — không còn key để mất. Mảnh cuối cùng là verify ở admission: cluster từ chối mọi image thiếu chữ ký hợp lệ:

# Kyverno: chỉ cho chạy image đã được pipeline ký
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata: { name: verify-image-signature }
spec:
  validationFailureAction: Enforce
  rules:
  - name: require-signed-images
    match: { any: [{ resources: { kinds: [Pod] } }] }
    verifyImages:
    - imageReferences: ["registry.example.com/*"]
      attestors:
      - entries:
        - keys:
            publicKeys: |-
              -----BEGIN PUBLIC KEY-----
              ...nội dung file cosign.pub...
              -----END PUBLIC KEY-----

Ghép tất cả thành chuỗi khép kín: build sinh provenance (SLSA) → sinh SBOM (Syft) → ký image (Cosign) → admission verify (Kyverno) → runtime giám sát (Falco). Mỗi mắt xích chặn đúng một nhóm rủi ro CICD-SEC ở trên — khi phỏng vấn, vẽ được chuỗi này ra giấy là một điểm cộng rất lớn.

Transit Gateway (TGW) là "router trung tâm" nối tất cả VPC. ATTT cấp độ 4 & data residency

"ATTT L4" trong tiêu đề sơ đồ là phân loại hệ thống thông tin cấp độ 4 theo Nghị định 85/2016/NĐ-CP — hệ thống quan trọng, yêu cầu giám sát, phân vùng, mã hóa và lưu log nghiêm ngặt (khớp với các khối Inspection, SIEM, WORM log). Option A (AWS Local Zone Hà Nội) và Option B (DX/VPN về CMC/Viettel) là hai phương án đặt dữ liệu tại Việt Nam để đáp ứng yêu cầu lưu trữ dữ liệu trong nước — lưu ý sơ đồ ghi đúng: Local Zone chỉ có 1 AZ vật lý, nên phải cân nhắc HA.

Hai luồng cần thuộc lòng khi nhìn sơ đồ

Luồng traffic người dùng (bắc-nam)

  1. User → Cloudflare (CDN, WAF chặn OWASP Top 10, chống DDoS/bot).
  2. → Public ALB trong Ingress VPC (điểm chạm Internet duy nhất).
  3. → TGW ép qua Inspection VPC (firewall soi sâu gói tin).
  4. → Workload VPC, vào tầng LB subnet → App subnet (Pod trên EKS).
  5. App gọi Cache/DB ở tầng subnet sâu hơn; gọi Core Trading Engine qua PrivateLink + mTLS; đi Internet chiều ra phải vòng qua Egress VPC (NAT) và bị re-inspect.
  6. Admin không SSH thẳng: vào qua Cloudflare Access (Zero Trust — xác thực danh tính từng request thay vì tin VPN).

Luồng CI/CD (dải dưới cùng của sơ đồ — khớp chương 07 + 09)

  1. Dev push code → App Source Repo → trigger Jenkins/CodePipeline: build + unit test.
  2. SAST + SCA (CodeQL/Semgrep + quét dependency) — fail là dừng.
  3. Build container image → đẩy lên ECR, registry tự scan + sign.
  4. CI commit tag image mới vào GitOps Config Repo (nhánh dev/test/stg/prod).
  5. Argo CD Hub (Shared Services) thấy thay đổi → sync xuống Spoke Non-Prod (tự động).
  6. Staging chạy → DAST (ZAP/Burp) quét URL staging.
  7. Pass → Approval Gate (PR review + duyệt tay) → merge sang nhánh prod.
  8. Argo CD sync Prod EKS trong sync window có kiểm soát.

13Infrastructure as Code

Terraform · Ansible · Cloud

Terraform — dựng hạ tầng bằng code

Thay vì click trên console AWS, bạn khai báo hạ tầng trong file .tf; Terraform tính toán cần tạo/sửa/xóa gì để thực tế khớp khai báo. Lợi ích với DevSecOps: hạ tầng review được qua PR, quét được bằng Checkov trước khi tồn tại, và tái tạo được y hệt cho dev/staging/prod.

# main.tf — VPC + subnet tối giản (phiên bản thu nhỏ của Ingress VPC trong sơ đồ)
provider "aws" { region = "ap-southeast-1" }

resource "aws_vpc" "ingress" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  tags = { Name = "ingress-vpc", Env = "lab" }
}

resource "aws_subnet" "public_a" {
  vpc_id            = aws_vpc.ingress.id
  cidr_block        = "10.0.1.0/24"
  availability_zone = "ap-southeast-1a"
  tags = { Name = "public-az-a", Tier = "public" }
}
terraform init      # tải provider
terraform plan      # xem TRƯỚC sẽ thay đổi gì — luôn đọc kỹ output này
terraform apply     # thực thi
terraform destroy   # dọn sạch (quan trọng khi lab để khỏi tốn tiền)
checkov -d .        # quét chính code Terraform này trong CI
Cẩn thận

File terraform.tfstate chứa toàn bộ thông tin hạ tầng kể cả dữ liệu nhạy cảm. Không commit vào Git — lưu remote backend (S3 + DynamoDB lock) có mã hóa và giới hạn quyền đọc.

Ansible — cấu hình máy chủ tự động

Terraform dựng máy, Ansible cấu hình bên trong máy (cài gói, sửa config, hardening). Không cần agent — chỉ cần SSH. Với DevSecOps, Ansible hay dùng để áp CIS Benchmark hardening hàng loạt. Học: playbook, inventory, role, idempotency (chạy 2 lần kết quả như 1 lần).

AWS tối thiểu cho DevSecOps

  • IAM: user/role/policy, STS assume-role, nguyên tắc least privilege — nền của mọi thứ.
  • VPC networking: như chương 12.
  • EC2, EKS, ECR, S3, KMS, Secrets Manager, CloudWatch, CloudTrail — đủ để vận hành pipeline và đọc sơ đồ.
  • Lab miễn phí phần lớn dịch vụ với AWS Free Tier; luôn đặt billing alarm trước khi bắt đầu.
